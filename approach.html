Marker Detection - Approach Document

Alemeno Frontend Internship Assignment  |  Sanidhya Sishodia  |  April 2026

What I Built

A React Native Android app that uses the phone camera to scan a custom visual marker, extract it, correct its orientation, and display 20 captured frames in a results grid. The detection and image processing runs fully in native Kotlin using OpenCV so it is fast and doesn't block the UI.

I used Marker 1 from the provided set - a 140x140mm square with a thick solid black border and a 20x20mm solid black orientation square placed at one inner corner.

Why This Marker

Marker 1 is simple, unambiguous, and fast to detect. The thick black border gives a very strong contour signal even in imperfect lighting. The single corner dot is a reliable orientation cue - its position (top-left, top-right, bottom-right, or bottom-left) tells you exactly how to rotate the extracted image back to canonical form. There is no ambiguity in any of the four rotations.

Detection Pipeline

Every camera frame goes through this sequence in a background thread:

Photo JPEG
→ Bitmap → RGBA Mat → Grayscale
→ Downsample to max 1500px (speed - full res not needed for finding a quad)
→ Gaussian blur 5x5 (smooth out noise before thresholding)
→ Adaptive threshold, inverted (dark border regions become white)
→ Morphological close 3x3 (bridge small gaps in the border contour)
→ findContours, outermost only
→ For each contour:
    area filter (0.5% - 70% of frame)
    approxPolyDP → must give exactly 4 vertices
    convexity check
    aspect ratio 0.6 - 1.7 (roughly square)
    perspectiveWarp to 300x300
    → verifyMarker1Structure
→ If valid: extract from full-res gray, correctOrientation, encode JPEG base64

Why Adaptive Thresholding

A global threshold (like Otsu) picks one brightness cutoff for the whole image. If one side of the marker is in shadow and the other is brightly lit, a single threshold will either blow out the bright side or miss the dark side entirely.

Adaptive thresholding computes a local threshold for each pixel based on its neighborhood. The marker border is dark relative to whatever surrounds it, and that local contrast is preserved regardless of overall lighting. This is the single biggest robustness improvement over a naive approach.

Marker Verification

Finding a quad-shaped contour is not enough - phones see lots of rectangular things (books, screens, windows, tiles). The verifyMarker1Structure function warps the candidate to 300x300 and applies two checks:

  • Border check: All four edge strips of the warped image must have a low average brightness (mean below 140). This confirms the thick black border is present on all sides.
  • Orientation dot check: The four inner corner regions are sampled. The darkest corner is identified. It must have a mean below 110 AND must be at least 30 brightness units darker than the next-darkest corner. A solid black square is distinctly darker than any image content - this gap reliably separates the real dot from ordinary dark image content in other corners.

Using a relative comparison (dot vs. second-darkest) rather than absolute cutoffs makes the check robust to varying screen brightness, camera exposure, and any image printed or displayed inside the marker.

Orientation Correction

Once the dot corner is identified, the rotation needed to bring it to the top-left is straightforward:

Dot positionRotation applied
Top-leftNo rotation
Top-right90 degrees counter-clockwise
Bottom-right180 degrees
Bottom-left90 degrees clockwise

The rotation is applied after extracting the marker from the full-resolution grayscale image (not the downsampled version), so the final 300x300 output is sharp.

Performance

Detection runs on the downsampled image (capped at 1500px on the longest side) so the contour-finding and thresholding steps are fast. Once a valid marker is found, the perspective warp for extraction is done on the full-resolution grayscale, giving a clean 300x300 crop. Frames are captured every 250ms with a busy-flag preventing concurrent processing, so there is no queue buildup.

What I Would Improve With More Time

The current pipeline re-runs the full contour search on every frame. A natural improvement would be to track the marker across frames once detected - using the previous frame's corner positions as a tight region of interest - to reduce per-frame work. I would also add a visual overlay on the live camera feed showing the detected quad in real time so the user gets immediate feedback.
